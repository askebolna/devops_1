## Описание

Тестовое задание на позицию Junior DevOps. Проект реализует развертывание отказоустойчивой и безопасной связки Nginx (Reverse Proxy) и Python Backend с использованием Docker и Docker Compose.

## Технологический стек

- Контейнеризация: Docker, Docker Compose (V2)
- Прокси-сервер: Nginx 1.29.6 (на базе Alpine Linux)
- Веб-сервер: Python 3.15 (на базе Alpine Linux)
- Скрипт: Bash
- OS: Debian 13

## Структура репозитория

- `backend/` — исходный код и Dockerfile веб-приложения
- `nginx/` — конфигурация и Dockerfile прокси-сервера
- `.dockerignore` — исключение лишних файлов из контекста сборки Docker’а
- `.env.example` — шаблон настроек окружения
- `.gitignore` — исключение лишних файлов из системы контроля версий
- `docker-compose.yml` — описание инфраструктуры
- `run_debian.sh` — скрипт для автоматического развертывания (протестировано на Debian)

## Быстрый запуск

### Требования

- Docker и Docker Compose
- Открытый и свободный 80 порт (стандартные настройки окружения)

### Инструкция

1. Клонируйте репозиторий.

2. При необходимости, измените настройки окружения: 

```
cp .env.example .env
```

3. Запустите скрипт для автоматического развертывания, с последующей проверкой работоспособности:

```
chmod +x run_debian.sh
./run_debian.sh
```

## Решения по архитектуре

### 1. Безопасность контейнеров

- Backend запускается под non-root пользователем. В случае взлома злоумышленник ограничен в своих действиях.

- Файлы `app.py` и `nginx.conf` имеют права 444 (read-only для всех пользователей) и принадлежат пользователю `root`. Без root-доступа файлы подменить невозможно.

- В `nginx.conf` отключена директива `server_tokens`, чтобы не раскрывать версию сервера и усложнить процесс поиска уязвимостей.

### 2. Изоляция сети

- Используется выделенная bridge-сеть Docker.

- Backend не пробрасывает порты на хост-узел, что исключает доступ к веб-приложению мимо reverse-proxy.

### 3. Отказоустойчивость

- Реализованы проверки `healthcheck` для обоих сервисов.

- Удален дефолтный конфиг `/etc/nginx/conf.d/default.conf`, дабы избежать неопределенное поведение Nginx.

- Использована зависимость `condition: service_healthy` в `depends_on`. Nginx не начнет развертывание, пока Python-сервер не подтвердит готовность. 

- Оба сервиса имеют ключ `restart: unless-stopped`, что позволяет автоматически поднимать их после ошибки, которая проявляется как `non-zero exit code` (за исключением ручной остановки).

- Версии образов не используют значение `latest`, что гарантирует идемпотентность образов.

### 4. Оптимизация образов (images)

- Использованы образы на базе Alpine Linux, что позволяет минимизировать размер конечных образов и уменьшить количество предустановленных утилит (минимизация поверхности атаки).

## Допущения

- Предполагается, что 80-й порт на хост-системе свободен. В случае конфликта портов значение можно изменить в переменной `NGINX_PORT` файла `.env`.

- Использован стандартный модуль Python `http.server`. Это позволяет избежать сторонних зависимостей (`Flask/FastAPI`) и минимизировать размер образа. Многостадийная сборка из-за этого решения становится неуместной.

- Скрипт автоматического развертывания и файл docker-compose.yml ориентированы на современные версии Docker Compose (V2).

- Тестирование проводилось в системе на основе Debian. В других дистрибутивах (например, CentOS) может потребоваться ручная настройка firewall’а или SELinux для корректного проброса трафика.

- Так как задание требует Nginx принимать запросы на 80 порт, а также все порты ниже 1024 в Linux защищены от non-root пользователя, я использую стандартную модель безопасности Nginx: master-процесс стартует от `root`, worker-процессы переключаются на non-root пользователя `nginx` и теряют привилегии.
